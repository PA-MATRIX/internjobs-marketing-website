    **Hard-block policy (SAFETY-POLICY-01..03):**

    **Canned reply for hard-blocked student SMS (SAFETY-RESPONSE-01) — exact text:**
    ```
    hey — couldn't process that one. try rephrasing?
    ```
    This is lowercase, uses an em-dash (—), no emojis, and matches the AGENT_VOICE rules from `workflows/student-inbound.mjs`. Do NOT alter this text.

    **For the /webhooks/photon block**, replace the `if (messageId && store?.pool) { ... }` section with:
    ```js
    if (messageId && store?.pool) {
      // v1.3 SAFETY-01: Pre-LLM screen BEFORE agent sees the text.
      // Hard-block on prompt_injection >= 0.8. Fail-open — Lakera downtime
      // must never block student SMS. Canned reply matches AGENT_VOICE.
      const screenResult = await screenMessage(
        inbound.text || "",
        process.env.LAKERA_GUARD_API_KEY || "",
      );
      const injectionScore = screenResult.score ?? 0;
      const isHardBlock = screenResult.flagged && injectionScore >= 0.8;

      // Log every non-passing result for /ops/safety (Plan 20-03 writes to safety_events).
      // For now: structured log that Plan 20-03 will also write to Neon.
      if (screenResult.action !== "passed") {
        console.log(JSON.stringify({
          level: "info",
          event: "lakera_screen",
          action: screenResult.action,
          reason: screenResult.reason,
          score: screenResult.score,
          channel: "sms",
          hard_block: isHardBlock,
          sender_last4: (inbound.channelAddress || "").slice(-4),
          message_id: messageId,
          preview: (inbound.text || "").slice(0, 80),
        }));
      } else {
        // Debug-level only on clean traffic (no log noise on normal SMS)
        console.debug(JSON.stringify({ level: "debug", event: "lakera_screen", result: "clean", message_id: messageId }));
      }

      if (isHardBlock) {
        // Send canned reply via smsProvider. Match AGENT_VOICE exactly.
        // No throw on send failure — log and continue so the 200 still goes
        // back to Photon (no retry storm).
        try {
          await smsProvider.sendSms(
            inbound.channelAddress,
            "hey — couldn't process that one. try rephrasing?",
          );
        } catch (sendErr) {
          console.error(JSON.stringify({
            level: "error",
            event: "safety_block_reply_failed",
            error: sendErr?.message ?? String(sendErr),
          }));
        }
        // Do NOT call runStudentInboundWorkflow — agent must not see this text.
        // The inbound_messages row is already written (audit trail preserved).
      } else {
        // Proceed to agent (soft-flag or clean)
        runStudentInboundWorkflow({
          pool: store.pool,
          messageId,
          smsProvider,
          config,
        }).catch((err) => {
          console.error(JSON.stringify({
            level: "error",
            message: "student_inbound_workflow_failed",
            messageId,
            error: err?.message ?? String(err),
          }));
        });
      }
    }
    ```

    Insert the Lakera screen gate in `apps/parrot/workers/lib/inbound-email.ts` inside `receiveEmail()`, BEFORE the `mailboxStub.createEmail()` call (currently around line 185).

    **Why before createEmail():** `createEmail()` triggers `EmployeeMailboxDO.extractTodosFromEmail()` as a fire-and-forget hook inside the DO. That's the LLM call we need to gate. Screening must happen before the DO sees the content.

    **Scope filtering (SAFETY-SCOPE-01..02 — critical):**
    Before calling `screenMessage`, check if the sender is a known `startup_member`. Known startup senders have already been authenticated by Cloudflare Email Routing and do not need Lakera screening. Screening them wastes quota and risks false positives on legitimate email.

    The `startup_members` table lives in Neon (apps/app, not in the Parrot Worker's DO SQLite). The Parrot Worker does NOT have direct Neon access. However, the `inbound-email.ts` handler DOES have the `WorkspaceDO` binding via `env.WORKSPACE`. Check if the sender is a known employee workspace email first — but for startup members, we need a different approach.

    **Practical approach for v1.3:** The Parrot Worker can query whether the sender's email matches an employee email (via `env.WORKSPACE.getEmployeeByWorkspaceEmail(sender)`). Startup members are harder to query from the Worker — they live in Neon. For v1.3, use a simpler heuristic: if the sender email domain matches the `startup_members` domain OR if the sender is already an identified employee, skip screening. The primary protection is: if `employee` resolves to a known Parrot employee, this is internal — skip. For all other senders (cold email), screen.

    Actually, the flow in `receiveEmail()` already does employee lookup (lines ~101-125). The `employee` variable is the resolved Parrot employee (the person the email is TO, not the sender). The sender is `parsed.from?.address`. For v1.3, screen all inbound email bodies EXCEPT:
    - Emails from the resolved employee's own workspace email (edge case: self-send)
    - Skip if `env.LAKERA_GUARD_API_KEY` is not set (fail-open, already handled in helper)

    The full startup_members exclusion (looking up sender in Neon) requires the Parrot Worker to have a Neon connection binding, which it does not. Defer to a KV-based allowlist approach: if the sender address is in `PARROT_FEATURE_FLAGS` KV (already bound) under key `safety_skip_senders`, skip screening. This lets Raj add specific startup emails without a code deploy.

    **Add import** at the top of `inbound-email.ts` (after existing imports):
    ```typescript

    ```

    **Insert screen gate** in `receiveEmail()` BEFORE the `mailboxStub.createEmail()` call (around line 178):
    ```typescript
    // v1.3 SAFETY-01: Pre-LLM screen before EmployeeMailboxDO.createEmail()
    // which triggers extractTodosFromEmail() → kimi-k2.6 LLM call.
    //
    // Scope discipline:
    //   - Internal Mattermost messages: NOT screened (polled separately, not in this path)
    //   - Known startup_members: check PARROT_FEATURE_FLAGS KV allowlist
    //     (key: "safety_skip_senders", value: comma-separated emails).
    //     Full Neon lookup deferred to v1.4 (Worker has no Neon binding).
    //   - Cold email (unknown sender): ALWAYS screened.
    //
    // Email hard-block policy (SAFETY-RESPONSE-02):
    //   No auto-reply on hard-block — out-of-office loop risk.
    //   Log to structured console; Plan 20-03 writes to safety_events.
    const emailBody = parsed.html || parsed.text || "";
    const senderEmail = (parsed.from?.address || "").toLowerCase();

    // Check KV allowlist for known startup senders (skip screening for them)
    let skipScreen = false;
    if (env.PARROT_FEATURE_FLAGS && senderEmail) {
      const skipList = await env.PARROT_FEATURE_FLAGS.get("safety_skip_senders").catch(() => null);
      if (skipList) {
        const allowedSenders = skipList.split(",").map((s: string) => s.trim().toLowerCase());
        skipScreen = allowedSenders.includes(senderEmail);
      }
    }

    if (!skipScreen && emailBody.length > 0) {
      const _screenStart = Date.now();
      const screenResult = await screenMessage(emailBody, env);
      const _screenMs = Date.now() - _screenStart;

      const injectionScore = screenResult.score ?? 0;
      const isHardBlock = screenResult.flagged && injectionScore >= 0.8;

      if (screenResult.action !== "passed") {
        console.log(JSON.stringify({
          level: "info",
          event: "lakera_screen",
          action: screenResult.action,
          reason: screenResult.reason,
          score: screenResult.score,
          channel: "email",
          hard_block: isHardBlock,
          sender: senderEmail.slice(-20), // partial — avoid PII in logs
          employee_id: employee.id,
          preview: emailBody.slice(0, 80),
        }));
      }

      console.log(JSON.stringify({ level: "debug", event: "lakera_latency_ms", ms: _screenMs, channel: "email" }));

      if (isHardBlock) {
        // Email hard-block: NO auto-reply (SAFETY-RESPONSE-02).
        // Out-of-office loop risk: if the blocked email is from an automated
        // sender, an auto-reply triggers their auto-responder, creating an
        // infinite loop. Operator reviews /ops/safety and replies manually.
        console.warn(JSON.stringify({
          level: "warn",
          event: "lakera_hard_block_email",
          employee_id: employee.id,
          reason: screenResult.reason,
          preview: emailBody.slice(0, 80),
        }));
        return; // Drop silently — no createEmail(), no todo extraction, no auto-reply
      }
      // Soft-flag: proceed to createEmail() normally. Event already logged above.
    }
    ```

