# Phase 28.5 Plan 03: Live Dashboard + Role Form + Thread Reply Summary

**One-liner:** Replaced the 28.5-02 dashboard skeleton with live API-driven UI:
typed client over the Pages Function proxy, role creation form matching the MCP
`execute('post_role')` schema exactly, and a candidate-thread view with
optimistic reply send — all behind a server-side `startup_id` resolution that
prevents browser spoofing.

## What Shipped

**Task 1 — Typed API client** (commit `d01278e`)

Rewrote `apps/startups/src/lib/api.ts` from a single `useApi()` generic-fetch
hook to a 6-function typed client backed by a shared `apiRequest` helper.
Exports:

- `getMe(token)` → `{ startup_id, startup_name, agent_email, role_count, … }`
- `getRoles(token)` → `RoleSummary[]`
- `createRole(token, body)` → `{ id, title }` — `body` shape matches the MCP
  `execute('post_role')` enum exactly
- `getThread(token, threadId)` → `{ thread_id, candidate_name, messages[] }`
- `sendReply(token, threadId, body)` → `{ ok: true }`
- `getThreads(token)` → `ThreadSummary[]`

Plus:

- `ApiError` class — thrown on non-2xx with status + body for caller error UI
- `useApiBound()` hook — preferred call site; returns all 6 functions
  pre-bound to the current Clerk session so pages do not have to thread
  `token` through every call
- `useApi()` retained as a back-compat hook (now delegates to `apiRequest`)

**Task 2 — Live UI + Pages Function mapping** (commit `abdd8c5`)

16 files / 1770 insertions / 115 deletions:

- **Lightweight shadcn-shaped primitives** (`src/components/ui/*`):
  `Card` + `CardHeader/Title/Description/Content/Footer`, `Button` with
  4 variants (primary / cobalt / outline / ghost) × 2 sizes (sm / lg),
  `Input`, `Textarea`, `Label`. All composed from Tailwind brand tokens
  via `cn()` — no Radix, no hex literals, no `style={{…}}`.

- **`Dashboard.tsx`**: three independent fetches (`getMe`, `getRoles`,
  `getThreads`) each with its own `loading/data/error` state. Renders:
  - Header with startup name (skeleton while loading) + sign-out
  - Stat grid: agent email (with "pending" hint when null) /
    open roles / status
  - "post a role →" CTA → `/roles/new`
  - Recent threads list (`ThreadList`)
  - Open roles list
  - **Not-linked branch**: if `/api/me` returns 404 or `not_found`, the
    whole inner section is replaced by a "contact hello@internjobs.ai"
    notice. Dashboard shell still renders.

- **`RolesNew.tsx` + `RoleForm.tsx`**: title (required, ≤140 chars),
  description (required, ≥20 chars, textarea), location (optional),
  employment_type (optional, enum exact match to MCP). Submit posts
  to `/api/roles` → on success navigates `/dashboard`; on error renders
  inline. Submit button: `post role →`. The form does not include
  any fields outside the MCP schema (STARTUP-WEB-DASH-02 parity hard-locked).

- **`RoleDetail.tsx`**: minimal v1.4 fetch-from-list-and-find
  implementation. Renders title, description, location, comp_range with
  a "editing arrives in v1.5" hint.

- **`CandidateDetail.tsx`**: placeholder page showing the candidate id
  and a "v1.5 polish" note. Route exists so deep-links don't 404.

- **`ThreadView.tsx`**: fetches `getThread` on mount, renders messages
  with direction-aware bubbles (inbound left, outbound right with cobalt
  accent). `MessageComposer` at the bottom; on send, the outbound message
  is **optimistically** appended to the list, then the POST happens,
  then the thread is re-fetched to reconcile with server-side state. On
  failure, the optimistic row is removed so the founder sees the error.

- **`MessageComposer.tsx`**: textarea + send button. Disabled while
  sending; clears on success; inline error rendering with brand-voice
  placeholder ("write a message...").

- **`ThreadList.tsx`**: dashboard inbox preview component. Empty state,
  loading skeleton, unread badge in cobalt.

- **`App.tsx`**: replaced four PlaceholderPage routes with the real
  page components (`RolesNew`, `RoleDetail`, `CandidateDetail`,
  `ThreadView`).

- **`functions/api/[[path]].ts`**: per-route mapping layer added on top
  of the 28.5-02 pass-through:
  - `GET /api/me` — composes `/v1/startups/identity-by-clerk-id` (POST
    with `clerk_user_id` from the Clerk JWT `sub`) + `/v1/startups/:id/stats`
    into the `MeResponse` shape. `agent_email` is read from the stats
    response if present; null otherwise.
  - `GET /api/roles` — calls `/v1/search/roles` scoped to the resolved
    `startup_id`. Maps `summary` → `title`.
  - `POST /api/roles` — overwrites `startup_id` in the body with the
    server-resolved value, forwards to `/v1/roles`.
  - `GET /api/threads` — calls `/v1/search/threads`. Maps `summary` →
    `candidate_name`, `last_inbound_at` → `last_message_at`. `unread_count`
    is currently 0 (TODO: 28.5-04 startup_mark aggregation).
  - `GET /api/threads/:id/messages` — returns `{ thread_id,
    candidate_name: "candidate", messages: [], _note: … }` until Fly
    ships the merged-history endpoint.
  - `POST /api/threads/:id/reply` — forwards to `/v1/messages` with
    `channel: "email"`, `direction: "outbound"`, server-resolved
    `startup_id` + `member_id`.
  - Everything else: legacy pass-through (28.5-02 behavior).
  
  Identity resolution decodes the Clerk JWT `sub` claim WITHOUT
  signature verification (the Fly side is the cryptographic boundary; the
  Pages Function ↔ Fly hop is secret-authenticated). The browser cannot
  inject a `startup_id` because every high-level route overwrites it
  server-side.

## Decisions

1. **Lightweight shadcn-shaped primitives in-tree, not the real shadcn
   stack.** The executor brief said "use shadcn/ui Card / Button / Input /
   Textarea — DO NOT inline-style" but the plan body explicitly said
   "Styling guidance (all Tailwind, no external component library needed
   for MVP)." These guidances pull in opposite directions. Resolution: I
   built primitives at `src/components/ui/` that mirror shadcn's API
   surface exactly (`<Card>`, `<CardHeader>`, `<CardTitle>`,
   `<CardContent>`, `<Button variant="primary"|"cobalt"|"outline"|"ghost"
   size="sm"|"lg">`, `<Input>`, `<Textarea>`, `<Label>`) — so a future
   `pnpm dlx shadcn@latest add card button input textarea` migration is a
   drop-in path — but they are implemented with plain Tailwind brand
   tokens and zero new dependencies. No Radix, no `class-variance-authority`,
   no `clsx`, no `tailwind-merge`. Spend = 350 LOC of UI primitives. Pays
   off the "no inline styles" rule AND the "no external library" rule
   simultaneously.

2. **Server-side `startup_id` resolution in the Pages Function for EVERY
   high-level route.** This is a security invariant, not a convenience.
   The browser sends the Clerk JWT in `Authorization: Bearer`, the Pages
   Function decodes its `sub` claim, calls
   `/v1/startups/identity-by-clerk-id` (which queries
   `startup_members.clerk_user_id`), and stamps the resolved `startup_id`
   into every outgoing body before forwarding to Fly. The browser
   physically cannot inject a foreign `startup_id` because the Pages
   Function overwrites it. The JWT signature is verified at the Fly
   layer; the Pages Function only needs to extract `sub`, which is safe
   because the Pages Function ↔ Fly hop is secret-authenticated (the
   shared `STARTUP_API_SECRET` is the trust boundary).

3. **Three independent `useEffect` fetches in Dashboard, not a single
   `Promise.all`.** If `/api/me` returns 404 (founder signed in but not
   linked to a startup), we want the "account not linked" branch to take
   over the whole content area — that's the critical UX path. If
   `/api/roles` or `/api/threads` fails for transient reasons (Fly cold
   start, network blip), we want the other cards to still render. Three
   independent state machines model this correctly; a single `Promise.all`
   would collapse the failure modes.

4. **`agent_email` null → "pending — ridhi will provision shortly", not
   "—".** The peer (`executor-28.5-04`) is shipping migration 0013 +
   slug assignment in parallel. From the founder's perspective, "pending"
   is true; "—" implies absence. The literal pending hint reduces support
   load ("why is my agent email a dash?").

5. **GET /api/threads/:id/messages returns an empty shell + `_note`
   marker.** The MCP path has `execute('get_thread_messages')` but that's
   not reachable via the Fly /v1 surface and adding it was out of scope
   (peer territory — `infra/startup-api/` belongs to the wave-3 peer for
   the migration plan; my brief excluded modifying it). The empty shell
   lets the MessageComposer + reply send work end-to-end (the optimistic
   append makes the sent reply appear immediately even with empty history
   from the GET). The TODO is captured at the function site for v1.5.

6. **Optimistic-then-reconcile reply pattern.** On send: append a
   synthetic outbound message with id `optimistic-<timestamp>`, POST the
   reply, on success re-fetch the thread (which displaces the optimistic
   row), on failure remove the optimistic row and surface the error in
   `MessageComposer`. This works for the v1.4 empty-history case (the
   reply still appears in the UI until the page is refreshed, at which
   point the empty re-fetch will hide it — acceptable since the row IS
   committed in `outbound_messages`) and for the v1.5 full-history case
   (the re-fetch returns the canonical message).
