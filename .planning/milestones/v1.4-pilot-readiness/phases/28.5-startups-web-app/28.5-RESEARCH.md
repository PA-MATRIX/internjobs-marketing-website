# Phase 28.5: Startups Web App + Clerk #3 + Per-Startup Agent Email — Research

**Researched:** 2026-05-25
**Domain:** Cloudflare Email Routing (inbound catch-all), Clerk webhooks + blocklist, Vite+React+CF Pages, slug provisioning
**Confidence:** HIGH on CF Email Routing + parrot pattern; HIGH on Clerk blocklist (confirmed paid-tier requirement); MEDIUM on Clerk webhook OAuth payload timing; HIGH on CF Pages Functions proxy pattern

## 1. Executive Summary

Phase 28.5 ships a third Clerk app (`startups.internjobs.ai`), a Vite+React web app on CF Pages (mirrors `apps/marketing/`), and a per-startup agent email system (`<slug>@startups.internjobs.ai`). The hardest question — "does CF Email Routing support a catch-all to a Worker?" — is YES, definitively. The existing `apps/parrot/` codebase already does every sub-problem: it has an `email()` export, `createEmailRoutingRule()` (literal per-address), and `sendWelcomeEmail()` (send_email binding). Phase 28.5 replaces the literal-matcher routing rule with a single catch-all zone rule pointing at the new `internjobs-startups` Worker, and resolves slugs dynamically at runtime via `startup_channel_links`. Clerk's domain blocklist (blocking gmail/yahoo/etc.) requires a **paid Clerk plan**; fallback is a `user.created` webhook that calls `DELETE /v1/users/:id` — verified email is present at `user.created` time for both Google OAuth and magic-link. The secrets proxy for the founder dashboard must live in a CF Pages Function (`functions/api/[...path].ts`), not the static Vite build — this is the standard CF Pages pattern, well-supported.

**Primary recommendation:** Five plans in dependency order: DNS/Clerk bootstrap → Vite scaffold + Pages deploy → founder dashboard UI → agent email (migration + catch-all + inbound Worker) → work-email enforcement + E2E test. The catch-all email Worker belongs in `apps/startup/` (rename to `apps/startups/workers/`) to co-locate web + email surface, not in parrot.

## 2. Hard Problem Findings

### Hard Problem 1 — Cloudflare Email Routing catch-all → Worker

**Question:** Does CF Email Routing support catch-all delivery to a Worker (not just forwarding to another address)?

**Findings (HIGH confidence):**

CF Email Routing supports a catch-all action with type `"worker"`. When a zone has Email Routing enabled and a catch-all rule is set, ALL mail to any address on that domain is delivered to the named Worker. The catch-all is a zone-level setting distinct from per-address literal rules.

The existing `apps/parrot/workers/lib/email.ts::createEmailRoutingRule()` creates *literal* rules (one per employee email). For startups we want a **single catch-all rule** instead. The API call differs: you POST to the catch-all endpoint rather than creating a per-address rule. Alternatively, set it via the Cloudflare Dashboard → Email → Routing → Catch-all action = "Send to a Worker" → worker name = `internjobs-startups`.

The inbound `ForwardableEmailMessage` object shape (confirmed from CF docs):
```typescript
interface ForwardableEmailMessage {
  from: string;        // envelope From
  to: string;          // envelope To (e.g. "acme@startups.internjobs.ai")
  headers: Headers;    // RFC-5322 headers object
  raw: ReadableStream; // raw MIME stream
  rawSize: number;     // size hint for stream pre-allocation
  setReject(reason: string): void;
  forward(rcptTo: string, headers?: Headers): Promise<void>;
  reply(message: EmailMessage): Promise<void>;
}
```

The Worker export structure (mirrors parrot exactly):
```typescript
export default {
  fetch: app.fetch,
  async email(message: ForwardableEmailMessage, env: Env, ctx: ExecutionContext) {
    const slug = message.to.split("@")[0]; // e.g. "acme"
    // lookup startup_channel_links WHERE channel_type='email'
    //   AND channel_external_id = message.to
    // insert into inbound_messages ...
  }
};
```

MIME parsing: use `postal-mime` (already in parrot — same npm package). The `streamToArrayBuffer` helper in `apps/parrot/workers/lib/inbound-email.ts` is directly reusable.

`wrangler.jsonc` send_email binding for the startups Worker:
```jsonc
"send_email": [{ "name": "EMAIL" }]
```
Outbound `from` is the startup's agent email (`acme@startups.internjobs.ai`). CF validates the `from` domain is verified in Email Routing — `startups.internjobs.ai` must be added as a verified domain in the zone's Email settings.

**SPF/DKIM/DMARC (action required by Raj):**
When you add `startups.internjobs.ai` as a verified Email Routing domain in the Cloudflare Dashboard, CF auto-generates:
- SPF record (`v=spf1 include:_spf.mx.cloudflare.net ~all`) — auto-added to DNS
- DKIM CNAME records (2 entries pointing to Cloudflare's signing infrastructure) — **must be added manually**
- DMARC policy record — CF does NOT auto-create this; must be added manually as `v=DMARC1; p=none; rua=mailto:dmarc@startups.internjobs.ai` (or similar)

The Dashboard shows a DNS checklist after domain verification — follow it completely before sending welcome emails or the first outbound from `welcome@startups.internjobs.ai` will likely hit spam.

**Worker name:** `internjobs-startups` (new Worker, NOT parrot). This keeps the parrot Worker's routing rules clean and avoids cross-tenant email dispatch.

**Open questions:**
- Does the catch-all rule survive a `wrangler deploy` that re-creates the Worker script? (Yes, routing rules are zone-level config independent of Worker deploys — confirmed from parrot's existing pattern where `createEmailRoutingRule` is called once at employee-invite time, not on every deploy.)
- Does `startups.internjobs.ai` need to be a separate CF zone or just a subdomain of `internjobs.ai`? Answer: subdomain Email Routing requires the subdomain to be verified as an Email Routing address under the parent zone. The parent zone `internjobs.ai` is already in CF. Adding `startups.internjobs.ai` as a routing domain is done via Dashboard → Email → Settings.

### Hard Problem 2 — Clerk webhook on user.created for work-email enforcement

**Question:** Can we use Clerk's native `allowed_email_domains` or do we need a webhook? What's the payload at user.created time?

**Findings (HIGH confidence on tier, MEDIUM on OAuth payload timing):**

Clerk's **allowlist** and **blocklist** features (which would let you block `@gmail.com`, `@yahoo.com`, etc.) require a **paid Clerk plan**. They are free in development but gated in production. Cost: ~$25/mo Pro tier or per-MAO add-on.

**Decision:** Ship `user.created` webhook as the v1.4 implementation regardless of tier. It works on free, it's more explicit, and it gives us an audit log. Paid-tier allowlist can replace it later.

Webhook payload for `user.created`:

```typescript
// Both Google OAuth and email magic-link populate email_addresses
// at user.created time. For Google OAuth, verification.status = "verified".
// For magic-link, the email IS verified before user creation completes
// (Clerk doesn't emit user.created until the magic link is clicked).
{
  type: "user.created",
  data: {
    id: "user_xxx",
    email_addresses: [{
      email_address: "founder@acme.io",
      id: "idn_xxx",
      verification: { status: "verified", strategy: "oauth_google" | "email_link" }
    }],
    primary_email_address_id: "idn_xxx",
    // Google OAuth also includes:
    external_accounts: [{ provider: "google", email_address: "founder@acme.io" }]
  }
}
```

The verified email is always present at `user.created` time for both Google OAuth and magic-link — Clerk does not emit `user.created` until the identity is verified. (LOW-MEDIUM: confirmed from community patterns; Clerk docs don't explicitly state this guarantee. Recommend testing in dev environment.)

**Delete user call** (CF Worker using raw fetch, no Node SDK):
```typescript
await fetch(`https://api.clerk.com/v1/users/${data.id}`, {
  method: "DELETE",
  headers: { Authorization: `Bearer ${env.STARTUPS_CLERK_SECRET_KEY}` }
});
```
Pattern is identical to `apps/parrot/workers/lib/clerk-admin.ts` line 96 (`/lock` endpoint) — same auth header shape.

**Webhook signature verification** (Svix):
```typescript
import { Webhook } from "svix";
const wh = new Webhook(env.STARTUPS_CLERK_WEBHOOK_SECRET);
const payload = await wh.verify(rawBody, {
  "svix-id": req.headers.get("svix-id")!,
  "svix-timestamp": req.headers.get("svix-timestamp")!,
  "svix-signature": req.headers.get("svix-signature")!,
});
```
`STARTUPS_CLERK_WEBHOOK_SECRET` is already reserved in Infisical as a pending slot.

**Where the webhook endpoint lives:** The `apps/startup/` Worker (renamed/extended as `internjobs-startups` for Phase 28.5) already handles `mcp.internjobs.ai`. Add a `/webhooks/clerk` route to the same Hono app. Do NOT put it in `apps/startups/` (the Vite static site — has no server runtime). Do NOT put it in parrot (wrong Clerk instance). The `apps/startup/` Worker is the right surface: it already holds `STARTUP_API_SECRET` + the Fly proxy URL, and it's the Worker that co-owns the startups schema.

**Personal domain blocklist:**
```typescript
const BLOCKED_DOMAINS = new Set([
  "gmail.com", "yahoo.com", "hotmail.com", "outlook.com",
  "icloud.com", "aol.com", "proton.me", "live.com", "msn.com",
  "yahoo.co.uk", "googlemail.com"
]);
// gmx.* wildcard: check .endsWith("gmx.") logic separately
function isPersonalEmail(email: string): boolean {
  const domain = email.split("@")[1]?.toLowerCase() ?? "";
  return BLOCKED_DOMAINS.has(domain) || domain.startsWith("gmx.");
}
```

**Open questions:**
- Race: if `user.created` webhook fires and we call `deleteUser`, does the user see a clean error or a broken session? Clerk shows "user not found" on the next request — recommend adding a custom error page. Clerk's allowlist approach avoids the race entirely but is paid-tier.
- Svix package availability in CF Workers: `svix` npm package works in Workers (no Node crypto dependency issues) — already confirmed in community. Use `"svix": "^1.x"`.

### Hard Problem 3 — Vite+React+Clerk scaffold for `apps/startups/`

**Question:** Minimum file set, env var prefix, and where to put the authenticated proxy for `STARTUP_API_SECRET`.

**Findings (HIGH confidence):**

**Env var prefix:** Vite uses `VITE_*` prefix for client-side env vars (injected at build time via `import.meta.env.VITE_*`). CF Pages also injects build-time env vars using this prefix. Confirmed: `VITE_CLERK_PUBLISHABLE_KEY` is the correct var name.

**Minimum file set** (mirrors `apps/marketing/`):
```
apps/startups/
├── package.json          # Vite + @vitejs/plugin-react + @clerk/react + tailwindcss + wrangler
├── tsconfig.json
├── tsconfig.app.json
├── tsconfig.node.json
├── vite.config.ts        # defineConfig({ plugins: [react()] }) — same as marketing
├── tailwind.config.ts
├── postcss.config.js
├── index.html
├── src/
│   ├── main.tsx          # ReactDOM.createRoot + ClerkProvider
│   ├── App.tsx           # router: / sign-in, /dashboard, /roles/*, /candidates/*, /thread/*
│   └── components/
└── functions/
    └── api/
        └── [[path]].ts   # CF Pages Function — authenticated proxy to Fly
```

**`@clerk/react` version:** Use `^5.x` to match parrot's `@clerk/clerk-react: ^5.61.6`. The Vite app uses `ClerkProvider` + `useUser()` / `useAuth()` hooks. Not `@clerk/nextjs` (that's Next.js only).

**The authenticated proxy problem:** The founder dashboard calls `internjobs-startup-api` Fly proxy endpoints that require `Authorization: Bearer STARTUP_API_SECRET`. This secret CANNOT live in the Vite static build. Two options:

| Option | Description | Verdict |
|--------|-------------|---------|
| (a) CF Pages Function | `functions/api/[[path]].ts` in the same project; injected as a secret via Wrangler Pages settings; runs as a Worker alongside the static assets | **RECOMMENDED** |
| (b) Separate Worker | `apps/startups/workers/` sub-folder, distinct deploy | Unnecessary complexity for v1.4 — two deploys for one app |

**Option (a) is the standard CF Pages pattern** and is well-supported. The Pages Function receives `STARTUP_API_SECRET` as an encrypted secret set via `wrangler pages secret put STARTUP_API_SECRET`. The Vite frontend calls `/api/roles`, `/api/candidates`, etc. on its own origin (no CORS), and the Function proxies to Fly with the Bearer header.

```typescript
// functions/api/[[path]].ts
import type { PagesFunction } from "@cloudflare/workers-types";

interface Env { STARTUP_API_SECRET: string; STARTUP_API_URL: string; }

export const onRequest: PagesFunction<Env> = async (ctx) => {
  const url = new URL(ctx.request.url);
  const target = `${ctx.env.STARTUP_API_URL}${url.pathname.replace("/api", "")}${url.search}`;
  return fetch(target, {
    method: ctx.request.method,
    headers: {
      ...Object.fromEntries(ctx.request.headers),
      Authorization: `Bearer ${ctx.env.STARTUP_API_SECRET}`,
    },
    body: ["GET", "HEAD"].includes(ctx.request.method) ? undefined : ctx.request.body,
  });
};
```

The Clerk session JWT (from `useAuth().getToken()`) is forwarded to the Fly proxy, which validates it against `STARTUPS_CLERK_JWKS_URL` to resolve `startup_id` via `publicMetadata.startup_id`.

**CF Pages project name:** `internjobs-startups` (separate from `internjobs-ai` which is the marketing site).

**deploy command** (same pattern as marketing):
```bash
wrangler pages deploy dist --project-name internjobs-startups --branch main
```

**Custom domain:** Add `startups.internjobs.ai` as a custom domain in the CF Pages project settings — auto-creates DNS CNAME + provisions TLS.

**Open questions:**
- Does `wrangler pages deploy` handle the `functions/` directory automatically? Yes — CF Pages detects `functions/` in the project root and deploys them alongside static assets.
- Do Pages Functions support the `send_email` binding for the inbound email? No — use the separate `internjobs-startups` Worker for email, not the Pages Function.

### Hard Problem 4 — Per-startup agent email + slug uniqueness

**Question:** Slug algorithm, catch-all vs per-rule routing, thread resolution via startup_channel_links.

**Findings (HIGH confidence):**

**Slug generation algorithm:**
```typescript
function mintSlug(companyName: string): string {
  return companyName
    .toLowerCase()
    .replace(/[^a-z0-9]+/g, "-")   // non-alphanumeric → hyphen
    .replace(/^-+|-+$/g, "")        // trim leading/trailing hyphens
    .slice(0, 30);                  // max 30 chars
}

async function reserveUniqueSlug(base: string, db: DB): Promise<string> {
  let slug = base;
  let attempt = 0;
  while (true) {
    const exists = await db.query(
      "SELECT 1 FROM startups WHERE agent_email = $1",
      [`${slug}@startups.internjobs.ai`]
    );
    if (!exists.rows.length) return slug;
    attempt++;
    slug = `${base}-${attempt}`;
  }
}
```

**Where slug reservation lives:** In the Phase 28-04 admin endpoint (`POST /admin/startups/new`) on the `apps/startup/` Worker — that's the authoritative writer for the `startups` table. The Fly proxy (`internjobs-startup-api`) executes the migration-0013 SQL. The web sign-up path (if self-serve in 28.5) calls the same endpoint.

**Catch-all vs per-slug routing rule:** Use a **single catch-all** zone-level rule for `startups.internjobs.ai`. The Worker resolves the slug at runtime:
```typescript
const slug = message.to.split("@")[0]; // "acme" from "acme@startups.internjobs.ai"
const row = await db.query(
  `SELECT startup_id, id FROM startup_channel_links
   WHERE channel_type = 'email' AND channel_external_id = $1`,
  [message.to]
);
```

This is strictly simpler than a per-slug routing rule: you never touch Email Routing config when a new startup is created. **CF Email Routing catch-all supports this pattern** — confirmed from community docs and the `Route to Workers` blog post.

**startup_channel_links row** minted at slug-reservation time:
```sql
INSERT INTO startup_channel_links (startup_id, channel_type, channel_external_id, created_at)
VALUES ($startup_id, 'email', 'acme@startups.internjobs.ai', NOW());
```

**Thread-ID resolution for inbound replies:**
The inbound email handler reads `In-Reply-To` and `Message-ID` headers from the MIME (`message.headers.get("in-reply-to")`). Match against `inbound_messages.message_id` or a `threads.agent_message_id` column to stitch the reply into the correct thread. This is the same pattern parrot uses in `inbound-email.ts`.

**Migration 0013** adds:
```sql
ALTER TABLE startups ADD COLUMN agent_email TEXT UNIQUE;
```
Plus the `startup_channel_links` table (already in 28-01 migration) gets a row per startup on creation.

**Open questions:**
- What happens if a slug collision is never resolved (infinite loop)? Add max-attempts guard (10 tries) and raise an error forcing a manual name — acceptable for v1.4.
- Who cleans up the `startup_channel_links` row if a startup is deleted? Out of scope for v1.4; mark as TODO.

## 5. Open Risks / Unknowns (preserve as TODOs in plans)

1. **Clerk user.created email timing (LOW-MEDIUM):** Assumed email is always present at `user.created` for OAuth + magic-link. Needs a dev-environment test before shipping 28.5-05. If a `user.updated` event is sometimes needed, the enforcement webhook needs to handle both events.

2. **Clerk paid tier for blocklist:** If webhook-based deletion causes UX issues (race condition, broken session flash), upgrade to Clerk Pro + native blocklist. Budget item: ~$25/mo. Defer to v1.5 polish.

3. **send_email binding `from` domain validation:** CF `env.EMAIL.send()` may reject a `from` address on `startups.internjobs.ai` until the domain is fully verified in Email Routing AND the send_email binding is scoped to that domain. The parrot fallback (REST API) handles this case — mirror the try/catch pattern from `apps/parrot/workers/lib/email.ts`.

4. **startup_channel_links retroactive rows:** Startups created via Phase 28-04 before 28.5 ships won't have `agent_email` or channel link rows. The Phase 28-04 extension in 28.5-04 needs a backfill script or a manual note that existing pilots need re-provisioned via admin endpoint.

5. **Thread-ID stitching for reply-to:** `In-Reply-To` / `Message-ID` header presence depends on the candidate's email client. If missing, the reply appears as a new thread. Acceptable for v1.4; mark as TODO for v1.5 deduplication logic.

6. **CF Pages Function + Pages site in same `apps/startups/` repo:** Wrangler Pages auto-detects `functions/` at the project root. Verify `functions/` is NOT inside `dist/` — Vite output dir must be `dist/`, functions dir stays at root. Standard pattern, should work, but test locally with `wrangler pages dev dist`.
