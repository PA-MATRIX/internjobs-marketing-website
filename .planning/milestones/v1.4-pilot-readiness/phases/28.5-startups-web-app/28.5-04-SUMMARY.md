# Phase 28.5 Plan 04: Per-Startup Agent Email Summary

**One-liner:** Per-startup `<slug>@startups.internjobs.ai` agent address ships
code-complete: migration 0013 (`startups.agent_email TEXT UNIQUE`), a 16-test
slug algorithm with deterministic mint + collision-safe HTTP-probe reservation,
a catch-all `email()` Worker handler that resolves slug → startup via
`startup_channel_links`, and `POST /admin/startups/new` extended to mint the
agent email + send a Clerk invite + send a welcome email — all behind four
new Fly proxy endpoints. Live deploy + Clerk secret bind + migration apply
deferred to `DEFER-28.5-04-A..C` per the "don't wait on me" rule.

## What Shipped

**Task 1 — Migration 0013, slug algorithm, wrangler.jsonc doc** (commit `bc33973`)

6 file changes — foundation layer:

- `apps/app/db/migrations/0013_v1_4_startup_agent_email.sql` (new): idempotent
  `ALTER TABLE startups ADD COLUMN IF NOT EXISTS agent_email text UNIQUE` +
  named partial index `startups_agent_email_idx WHERE agent_email IS NOT NULL`.
  Matches the 0011/0012 idempotency pattern; will be picked up by the existing
  `apps/app/scripts/migrate.mjs` runner in sequence.

- `apps/startup/workers/lib/slug.ts` (new, 102 LOC): two exports.
  - `mintSlug(companyName: string)`: pure, deterministic. Lowercase → strip
    non-alphanumeric → collapse hyphen runs → trim → slice to 30 chars → re-trim.
  - `reserveUniqueSlug(base, apiUrl, apiSecret)`: HTTP loop against the Fly
    proxy `GET /v1/startups/check-slug?agent_email=<addr>`. 404 = free →
    return. 200 = taken → advance. Re-truncates `<base>-<n>` to MAX_SLUG_LEN
    on collision (keeps email local-part within RFC-5321's 64-octet bound;
    we cap at 30, well under). Throws on max-attempts exhaustion (10), empty
    base, or non-2xx/404 HTTP status. Bearer-auth headers + 8-second
    AbortSignal timeout.

- `apps/startup/workers/lib/slug.test.ts` (new, 218 LOC): 16 unit tests via
  Node 22's built-in `node:test`. mintSlug (9 cases: two-word, whitespace
  collapse, punctuation, length cap, determinism, unicode strip, empty,
  numeric-only, dangling-hyphen-after-slice). reserveUniqueSlug (7 cases:
  first-404, collision-advance to -1/-2, max-attempts exhaustion, non-2xx
  error, empty-base guard, Bearer header + AbortSignal, long-base-with-
  collision length safety). Run with `npx tsx --test workers/lib/slug.test.ts`
  — all 16 pass in ~150ms.

- `apps/startup/workers/types.ts`: added `EMAIL?: SendEmail` and the four
  `STARTUPS_CLERK_*?: string` env bindings. All optional at type level so
  code paths guard at runtime.

- `apps/startup/wrangler.jsonc`: extended the existing `send_email` binding's
  doc comment to mention the 28.5-04 outbound path
  (`welcome@startups.internjobs.ai`) and the inbound catch-all `email()`
  export. No structural change — the binding was already declared by 28-05.

- `apps/startup/tsconfig.json`: exclude `workers/**/*.test.ts` so the
  Workers `tsc --noEmit` doesn't see `node:test` / `node:assert` imports.

**Task 2 — Inbound email handler + admin endpoint extension + Fly endpoints** (commit `0347803`)

6 file changes — runtime wiring:

- `apps/startup/workers/routes/email.ts` (new, 289 LOC):
  `handleInboundEmail(message, env, ctx)` — the catch-all CF Email Routing
  handler. Five-step flow:
  1. Extract slug from `message.to.split("@")[0]`; setReject on malformed.
  2. Resolve `startup_channel_links` row via Fly `GET /v1/channels/resolve`;
     setReject on 404, silently drop on infra failure (Fly downtime
     bouncing mail is worse than recovering from logs).
  3. Drain the raw MIME stream into an ArrayBuffer (postal-mime requires
     buffered input). Use `rawSize` hint for one-shot allocation.
  4. Parse In-Reply-To + References + Message-ID headers for thread
     stitching (best-effort — packed into `inbound_messages.metadata.thread_anchor`).
  5. POST `/v1/messages/inbound` with provider='cloudflare-email',
     channel_type='email', body, and the threading metadata blob.
  Structured-JSON logging at every error point so `wrangler tail` shows the
  full failure trail. Mirror of `apps/parrot/workers/lib/inbound-email.ts`
  minus the Lakera screen + WorkspaceDO routing (founders' inbound surface
  is lower-risk than Parrot's employee mailbox).

- `apps/startup/workers/app.ts`: added the `email()` export on the Worker
  default export alongside `fetch`. Three lines of glue + an import.

- `apps/startup/workers/routes/admin.ts`: three new private helpers + a
  route-handler injection (~290 LOC added):
  - `provisionAgentEmail({startup_id, member_id, company}, env)`:
    synchronous. mintSlug → reserveUniqueSlug → PATCH `agent_email` →
    POST `channel-links`. Returns the chosen `<slug>@startups.internjobs.ai`
    so the route handler can surface it in the response body.
  - `sendClerkInvite({founder_email, startup_id}, env)`: POSTs
    `https://api.clerk.com/v1/invitations` with the founder email +
    `public_metadata.startup_id`. Silently skips if
    `STARTUPS_CLERK_SECRET_KEY` is unbound (DEFER-28.5-04-A) — emits a
    structured warn instead of throwing.
  - `sendWelcomeStartupEmail({founder_email, company, agent_email}, env)`:
    `env.EMAIL.send({from, to, subject, text})` with lowercase brand-voice
    body. Try/catch around the send so the welcome email failing (e.g.
    pre-DEFER-28.5-01-D when the domain isn't verified) doesn't break the
    admin endpoint — falls through to a log-only line with the full body
    text for manual resend.

  Route-handler injection: between `createStartup()` returning success and
  the existing SMS-send / response-return, the handler now (a) synchronously
  provisions the agent email, (b) fires the Clerk invite + welcome email
  via `ctx.executionCtx.waitUntil()`, (c) adds `agent_email` and
  `agent_email_error` fields to the response JSON. The original SMS + token
  + install-snippet flow is untouched.

- `apps/startup/package.json` + `package-lock.json`: added
  `postal-mime ^2.6.1` (same version as `apps/parrot`).

- `infra/startup-api/src/index.mjs`: four new endpoints, all behind the
  existing `/v1/*` Bearer middleware:
  - `GET /v1/startups/check-slug?agent_email=<addr>` → 200/404 probe.
  - `PATCH /v1/startups/:id/agent-email` → writes the column; 409 on UNIQUE
    violation (a parallel onboarding raced past check-slug) so the caller
    can re-mint.
  - `GET /v1/channels/resolve?email=<addr>` → returns `{startup_id,
    member_id}` from `startup_channel_links WHERE channel_type='email'`.
  - `POST /v1/messages/inbound` → INSERT inbound_messages with full
    threading + identity metadata folded into the jsonb column. Uses
    `ON CONFLICT (provider, provider_event_id) WHERE provider_event_id IS
    NOT NULL DO NOTHING` for RFC Message-ID dedupe.

## Decisions

1. **Migration path: `apps/app/db/migrations/`, not `infra/startup-api/
   migrations/`.** Plan frontmatter said `infra/...`; investigation showed
   that directory doesn't exist and the migrate.mjs runner at
   `apps/app/scripts/migrate.mjs` reads from `apps/app/db/migrations/`
   (where 0011 + 0012 already live). The plan's prose body (and the
   executor-prompt) correctly said `apps/app/db/migrations/` — followed
   the actual runner contract. Rule 1 frontmatter-drift fix.

2. **Fly endpoints landed in the single-file `infra/startup-api/src/index.mjs`,
   not the `routes/startups.ts` + `routes/admin.ts` split the frontmatter
   suggested.** The Fly proxy is a flat 884-line index.mjs; the `routes/`
   split doesn't exist. Followed the actual file layout — adding routes to
   the existing file keeps all 13 Fly endpoints discoverable in one place,
   which has been the file's convention through Phases 28-01..28-05 and
   28.5-02. A future refactor to a `routes/` split is a separate
   architectural decision.

3. **Tests run via `node:test` (Node 22 built-in), not vitest/jest.**
   `apps/startup` had zero test infrastructure; adding vitest would mean
   installing 100+ transitive deps to a Worker package. `node:test` is
   zero-dep, native in Node 22, and the project already has `tsx` available
   via `npx`. Excluded `*.test.ts` from the Workers tsconfig so
   `tsc --noEmit` doesn't see `node:test` / `node:assert` imports (the
   Workers `lib`/`types` lists don't include Node built-ins). Tradeoff: no
   `--watch` mode out of the box, but that's not load-bearing for a 16-test
   suite that runs in 150ms.

4. **`EMAIL?: SendEmail` on the Env interface (optional binding).** At
   runtime the binding is always present in deployed builds (declared in
   `wrangler.jsonc`), but optional typing lets local `wrangler dev` and
   defensive code paths (`routes/api.ts`'s existing emailBinding cast
   pattern, the new `sendWelcomeStartupEmail`'s `if (!emailBinding) → log`
   fallback) be cleanly type-safe. Mirrors `apps/parrot/workers/types.ts`'s
   treatment of `CLOUDFLARE_EMAIL_API_TOKEN`. Optional typing also
   self-documents the "binding may be intentionally absent during
   DEFER-28.5-01-D" runtime window.

5. **Agent email provisioning is synchronous; Clerk invite + welcome email
   are async via `ctx.waitUntil()`.** The agent email is what Ridhi needs in
   the response body to hand to the founder — three Fly roundtrips (~150-300
   ms) is acceptable for a concierge-pace endpoint. The Clerk + welcome
   email roundtrips don't surface anything to Ridhi and can run
   post-response without affecting UX.

6. **`env.EMAIL.send({from:{email,name}, to:[{email}], subject, text})`
   shape, not `new EmailMessage(from, to, subject).setContent(text)`.** The
   plan text suggested the `EmailMessage` constructor API; that API doesn't
   exist on the CF `send_email` binding — it's from a separate
   `cloudflare:email` module (used by the legacy Email Workers SDK, not the
   Email Service Workers API). `apps/parrot/workers/lib/email.ts` and
   `apps/agentic-inbox/workers/email-sender.ts` both use the
   `{from, to, subject, text}` object shape on `env.EMAIL.send()`.
   Followed the existing pattern. Rule 1 API-drift fix.

7. **Inbound MIME from/subject/body_text/body_html → `inbound_messages.metadata`
   jsonb, not first-class columns.** The 0003b/0004 schema for
   `inbound_messages` has a single `body` column. Adding columns for the
   email-specific fields is architectural (Rule 4) and would mean a 0014
   migration. The jsonb approach preserves the data for v1.5 promotion
   (one straightforward migration that pulls `metadata->>'subject'` into a
   `subject` column with a default-null backfill). No data loss; deferred
   schema churn.

8. **ON CONFLICT DO NOTHING on inbound_messages uses the partial UNIQUE
   index from 0003b.** `(provider, provider_event_id) WHERE
   provider_event_id IS NOT NULL` was already in place — the new
   `POST /v1/messages/inbound` endpoint leverages it for RFC Message-ID
   dedupe of resends. Inbound emails without a Message-ID header (rare,
   from malformed senders) can't dedupe but also can't conflict — they
   just insert as fresh rows. Acceptable: content-hash dedupe is a v1.5
   item if real-world spam volume justifies it.

9. **Three new admin.ts helpers are private (no `export`).** They live
   above the Hono router and are not part of the route module's public
   API. The 28.5-05 webhook handler (next plan) will need its own
   `deleteClerkUser` helper but shouldn't reach into admin.ts internals;
   keeping these private prevents accidental cross-route coupling. If a
   future caller does need them, promoting to `export` is a one-line
   change.

10. **No live deploy in this plan.** Per the "don't wait on me — finish all
    the phases" rule (user, 2026-05-25), all three deploy/runtime steps
    (Clerk secret bind, migration apply, Fly+Worker redeploy) are captured
    in PHASE-28.5-DEFERRED-OPS.md as DEFER-28.5-04-A..C with exact
    command-line invocations and acceptance criteria. Code is verified
    locally and ready to deploy when raj@ runs the ops cycle.
