# Phase 28.5 Plan 02: Vite Scaffold + Sign-In + Dashboard + Pages Function Summary

**One-liner:** First Clerk-on-Vite app in the repo — `apps/startups/` ships sign-in
landing, protected dashboard skeleton, and a `functions/api/[[path]].ts` Pages Function
that keeps `STARTUP_API_SECRET` out of the static bundle while forwarding Clerk JWTs to
the Fly proxy via a new `POST /v1/startups/identity-by-clerk-id` endpoint.

**Task 2 — Sign-in, App router, Dashboard, ProtectedRoute, Pages Function, identity endpoint** (commit `72a13cc`)

7 source files + 1 Pages Function + 1 Fly proxy edit:

- `src/main.tsx`: `ClerkProvider` wraps `BrowserRouter` + `App`. `VITE_CLERK_PUBLISHABLE_KEY` validated at startup — missing key logs an explicit console error rather than silently rendering a broken auth experience.
- `src/App.tsx`: React Router v6 with 6 routes — `/` (SignIn), `/dashboard`, `/roles/new`, `/roles/:id`, `/candidates/:id`, `/thread/:id`. Five gated routes plus a 404 fallback. Placeholder pages for the four routes that ship real content in 28.5-03.
- `src/components/ProtectedRoute.tsx`: `<SignedIn>{children}</SignedIn>` + `<SignedOut><RedirectToSignIn /></SignedOut>` — Clerk's idiomatic gate component.
- `src/pages/SignIn.tsx`: Clerk `<SignIn />` widget centered on lavender bg with brand-v1 micro-copy ("internjobs.ai — startups portal", lowercase ink text, cobalt accent-dot on `.ai`). Sign-up unified into the same widget (`signUpUrl="/"`). After auth → `/dashboard`.
- `src/pages/Dashboard.tsx`: skeleton with 3 cards (agent email, open roles, status), a "post a role →" CTA → `/roles/new`, sign-out button via `useClerk().signOut({ redirectUrl: "/" })`, "recent threads" empty state. On mount, calls `GET /api/me` via the `useApi` hook; gracefully falls through to placeholders if the endpoint 404s (expected during 28.5-02 since `/v1/me` doesn't exist yet on the Fly side — 28.5-03 lands it).
- `src/lib/api.ts`: `useApi()` hook — wraps `fetch` with Clerk JWT attachment (`Authorization: Bearer <jwt>`) and same-origin `/api/*` URL construction. Returns parsed JSON or throws on non-2xx.
- `functions/api/[[path]].ts` (CF Pages Function): catch-all proxy. Strips `/api` prefix, rewrites to `STARTUP_API_URL/v1/*`. **Swaps** the incoming `Authorization` (Clerk JWT) for `Authorization: Bearer ${STARTUP_API_SECRET}` (matches the Fly `verifyBearer` contract) and **forwards** the Clerk JWT separately as `X-Clerk-Token`. 503 if env unconfigured; 502 on upstream fetch failure. Only mirrors `Content-Type` from upstream — no header leakage.
- `infra/startup-api/src/index.mjs`: added `POST /v1/startups/identity-by-clerk-id` endpoint. Body `{ clerk_user_id }`, returns `{ startup_id, member_id, startup_name, role }` or 404. Joins `startup_members.clerk_user_id` → `startups.id` with the same `status IN ('active','onboarding')` guard as the existing token path. No new schema needed — the column already exists from Phase 28.

## Decisions

1. **`startup_members.clerk_user_id` is the identity-resolution key, not Clerk's `publicMetadata.startup_id`.** The research note suggested both options; we picked the simpler one because the column already exists from Phase 28 (the MCP onboarding path writes a `concierge:` placeholder there), the 28.5-05 webhook will UPDATE it to the real `user_*` id at signup time, and there's no Clerk-side roundtrip needed at every request. Single source of truth = Postgres.

2. **Clerk JWT travels as `X-Clerk-Token`, not as `Authorization: Bearer`.** The Fly proxy's `verifyBearer` middleware expects `Authorization: Bearer <STARTUP_API_SECRET>` — the existing service-auth contract from Phase 28. Repurposing the slot for the Clerk JWT would break that. Forwarding the Clerk JWT as a separate header is cheaper than rewriting the Fly auth layer, and it keeps the two concerns (service-auth vs end-user-identity) explicit in the request flow.

3. **Identity endpoint added in 28.5-02, not deferred to 28.5-03.** The plan said "you may need to ADD this endpoint" as optional scope. Shipping it now keeps the Pages Function ↔ Fly contract complete from day one — when 28.5-03 wires the dashboard's real data fetch, the route already responds. Zero downstream value to deferring.

4. **Deploy step → DEFER-28.5-02-A.** DEFER-28.5-01-C (CF Pages project + custom domain) is unresolved. Code is deploy-ready (build passes, bundle audited, env contract documented in `.env.example`); the actual `wrangler pages deploy` waits on the upstream ops checkpoint. Per user instruction "don't wait on me — finish all the phases" (2026-05-25), the plan ships code-complete + deploy-ready rather than blocking.

5. **`@clerk/clerk-react`, not `@clerk/react`.** The plan text referenced `@clerk/react` (which doesn't exist as a published package). Parrot's `package.json` uses `@clerk/clerk-react ^5.61.6` and the imports work as the plan's example code intended (`SignIn`, `SignedIn`, `SignedOut`, `RedirectToSignIn`, `useUser`, `useAuth`, `useClerk`, `ClerkProvider` all exported). Rule 1 fix — silent bug that would have caused `npm install` to fail.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] Wrong Clerk package name in plan text**

- **Found during:** Task 1 scaffold (`npm install` would have failed on `@clerk/react`)
- **Issue:** Plan referenced `@clerk/react` which is not a published npm package. The Vite-compatible Clerk SDK is `@clerk/clerk-react`.
- **Fix:** Used `@clerk/clerk-react ^5.61.6` (matches parrot's existing version). All import statements updated accordingly.
- **Files modified:** `apps/startups/package.json`, all `apps/startups/src/*.tsx` using Clerk
- **Commit:** `f49197f` + `72a13cc`

**2. [Rule 1 — Bug] Pages Function used wrong header for Fly auth**

- **Found during:** Task 2 Pages Function implementation
- **Issue:** Plan example used `X-Startup-Api-Secret: ${env.STARTUP_API_SECRET}` as the Fly-auth header, but the existing `infra/startup-api/src/index.mjs` `verifyBearer` middleware (Phase 28) reads `Authorization: Bearer <STARTUP_API_SECRET>`. Using the wrong header would have returned 401 on every Pages-Function-to-Fly call.
- **Fix:** Pages Function writes `Authorization: Bearer ${STARTUP_API_SECRET}` (matches the contract). The incoming Clerk JWT is forwarded separately as `X-Clerk-Token`.
- **Files modified:** `apps/startups/functions/api/[[path]].ts`
- **Commit:** `72a13cc`

**3. [Rule 3 — Blocking] Missing `ImportMetaEnv` typing blocked TS build**

- **Found during:** Task 1 build verification (`tsc -b` failed with `TS2339: Property 'env' does not exist on type 'ImportMeta'`)
- **Issue:** Vite's `import.meta.env.VITE_CLERK_PUBLISHABLE_KEY` is untyped in a fresh TS scaffold; without a `vite-env.d.ts` triple-slash reference, `tsc -b` blocks the build.
- **Fix:** Added `src/vite-env.d.ts` with `/// <reference types="vite/client" />` plus a local `ImportMetaEnv` interface declaring `VITE_CLERK_PUBLISHABLE_KEY`.
- **Files modified:** `apps/startups/src/vite-env.d.ts` (new)
- **Commit:** `f49197f`
