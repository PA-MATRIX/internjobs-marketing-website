**In workers/lib/mattermost.ts**, add:

- `createMmDirectChannel(mattermostUrl, token, userIds: [string, string])` — POST /api/v4/channels/direct with body `[userId1, userId2]`. Both user IDs must be included; the employee's own MM user_id is one of them. Returns MattermostChannel | null.

- `createMmGroupChannel(mattermostUrl, token, userIds: string[])` — POST /api/v4/channels/group with body `[...userIds]`. At least 3 user IDs (including the creator). Returns MattermostChannel | null.

- `getMmMyDirectChannels(mattermostUrl, token)` — GET /api/v4/users/me/channels?include_deleted=false. Returns MattermostChannel[] filtered to type "D" and "G" (direct and group DMs). This endpoint returns all channels the token user is a member of; filter client-side for DMs.

Note from research (Pattern 4): MM DMs are channels of type "D" (direct) or "G" (group). Creating or opening a DM between the same two users is idempotent — MM returns the existing channel if it already exists.

**In workers/index.ts**, add the following routes (all require `requireEmployeeMailbox`, all use employee PAT via `resolveEmployeeToken`):

```
GET  /api/chat/dms        — list existing DM channels (type D + G) for the employee
POST /api/chat/dms/direct — open/create a DM with one user; body: { mm_user_id: string }
POST /api/chat/dms/group  — open/create a group DM; body: { mm_user_ids: string[] } (2+ other users)
```

For GET /api/chat/dms: call getMmMyDirectChannels with the employee PAT. Also batch-resolve display names for the DM partners using getMmUsersByIds (bot token is fine for user resolution — it has read access). Return channels with an enriched `dm_partner_names: string[]` field computed from the participants.

For POST /api/chat/dms/direct: resolve the employee's mm_user_id from WorkspaceDO (getEmployeeToken), call createMmDirectChannel with `[employee.mmUserId, body.mm_user_id]`. Return the channel object.

For POST /api/chat/dms/group: validate userIds.length >= 2, add employee's own mm_user_id to the array, call createMmGroupChannel. Return the channel object.

After creating a DM, it can be used immediately for posting via the existing `POST /api/chat/posts` route (channel_id is the DM channel's id — no special handling needed since the route already accepts any accessible channel_id).

**DM section in secondary nav:**
- Separate section below the channels list with a header "Direct Messages" and a "+" new DM button.
- Fetches GET /api/chat/dms every 30s (separate useQuery from channels).
- Each DM item shows: avatar initials (computed from partner name) + partner name for type "D"; for type "G" group DMs show "Group: [name1], [name2]..." truncated to 20 chars.
- Clicking a DM item sets `activeChannelId` to that DM channel's id — the existing message fetch (GET /api/chat/channels/:id/posts) already handles any channel type including DMs.

**New DM user picker dialog:**
- Triggered by the "+" button in the DM section header.
- The picker fetches the list of employees from the existing bootstrap data's team member list. Since `/api/chat/bootstrap` returns `channels` but not users, add a separate GET `/api/chat/members` route in index.ts that calls `getMmUsersByIds` or `getMmTeamPublicChannels` ... actually use the existing `/api/chat/users` endpoint (POST with all known user IDs from a team member listing). Better: add GET `/api/chat/team-members` that returns all members of the team using the bot token (`GET /api/v4/teams/{team_id}/members` + batch user lookup).

Add `GET /api/chat/team-members` to index.ts:
- Use bot token (MATTERMOST_BOT_TOKEN) to GET /api/v4/teams/{teamId}/members (requires the team_id from bootstrap)
- Then batch-lookup user profiles via getMmUsersByIds
- Return `MmUser[]` list (filtered to exclude the requesting employee)
- Cache at the Worker response level with a 60s Cache-Control header

User picker dialog UI:
- Simple inline search input to filter the employee list by name.
- Checkboxes to select multiple employees (group DM) or click on one for direct DM.
- Two buttons: "Direct Message" (disabled if >1 selected), "Group DM" (disabled if <2 selected).
- On confirm: call POST /api/chat/dms/direct or /api/chat/dms/group via useMutation. On success, close dialog, invalidate DM list query, switch active channel to the new/existing DM channel.

Do NOT add any new npm packages. The user picker is a simple Tailwind-styled modal overlay using existing component patterns.
