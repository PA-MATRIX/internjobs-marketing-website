<objective>
Wave 0 — Per-user Mattermost PAT identity. This is the architectural unlock that all subsequent waves depend on. Today the Worker proxies ALL chat calls through a single bot identity (MATTERMOST_BOT_TOKEN), posting human messages with `parrot_author_*` props as a workaround. After this plan, every provisioned employee has their own Mattermost personal access token (PAT) stored in WorkspaceDO SQLite, and the Worker uses it to post as the real MM user.

Purpose: Without per-user identity, Waves 1-5 cannot implement channels-as-user, DMs (DMs require posting as the participant), or real-time WebSocket auth (the WS authentication_challenge must send the user's own token). This is the prerequisite for everything.

Output: WorkspaceDO migration #2 (mm_user_id + mm_access_token columns), mintMmUserToken + mmFetchAsUser helpers, updated /api/chat/posts using user PAT, backfill admin endpoint, 401-triggered re-mint, updated ChatPane post renderer.

IMPORTANT: `MM_SERVICESETTINGS_ENABLEUSERACCESSTOKENS=true` MUST be set on the internjobs-mattermost Fly app before PAT minting will work in production. This is done in Wave 5 (31-06). For local dev: set this in your local MM instance config first. Do not block this plan on the Fly secret — implement everything, note the dependency in code comments, and verify locally. Do NOT call POST /api/admin/chat/backfill-tokens against the production Worker until plan 31-06 Task 1 confirms the Fly secret is present.
</objective>

**`mintMmUserToken`** — mints a PAT for a specific MM user using the admin token:
```typescript
export async function mintMmUserToken(
  mattermostUrl: string,
  adminToken: string,
  mmUserId: string,
): Promise<string | null> {
  const resp = await mmFetch<{ token: string }>(
    mattermostUrl,
    adminToken,
    `/api/v4/users/${mmUserId}/tokens`,
    { method: "POST", body: JSON.stringify({ description: "parrot-workspace" }) },
  );
  return resp.ok ? resp.data.token : null;
}
```

NOTE: This requires MM_SERVICESETTINGS_ENABLEUSERACCESSTOKENS=true on the Fly MM app (set in Wave 5 / plan 31-06). In local dev, ensure your local MM instance has this enabled. If it returns a 501 with `id: "api.user.create_user_access_token.disabled.app_error"`, the Fly secret is missing — do NOT proceed with production backfill until plan 31-06 Task 1 is complete.

**`mmFetchAsUser`** — proxy helper that resolves the employee PAT from WorkspaceDO and uses it for MM REST calls. Must accept `getTokenFn` as a dependency injection parameter (not import WorkspaceDO directly, to keep mattermost.ts free of DO coupling):

```typescript
export async function mmFetchAsUser<T>(
  mattermostUrl: string,
  adminToken: string,
  path: string,
  init: RequestInit,
  getToken: () => Promise<{ mmUserId: string; token: string } | null>,
  setToken: (mmUserId: string, token: string) => Promise<void>,
): Promise<{ ok: true; data: T } | { ok: false; status: number; data: unknown }> {
  const tokenRow = await getToken();
  if (!tokenRow) return { ok: false, status: 503, data: { error: "chat_not_provisioned" } };

  let result = await mmFetch<T>(mattermostUrl, tokenRow.token, path, init);

  // 401-triggered re-mint: PAT may have been revoked by an MM admin
  if (!result.ok && result.status === 401) {
    const newToken = await mintMmUserToken(mattermostUrl, adminToken, tokenRow.mmUserId);
    if (newToken) {
      await setToken(tokenRow.mmUserId, newToken);
      result = await mmFetch<T>(mattermostUrl, newToken, path, init);
    }
  }

  return result;
}
```

Note: `mmFetch` is currently a private function in this file. Make it exported OR expose it as `mmFetchRaw` for use in the WS proxy in Wave 4. Prefer exporting `mmFetch` directly (rename from the private function by adding `export`).
