CRITICAL implementation note (from research — do NOT buffer the file upload): The Worker must NOT call `c.req.formData()` for the file upload route. Use `c.req.raw.body` (the ReadableStream) as the body passed directly to the upstream MM fetch. Buffering a 50MB upload in Worker memory will cause OOM errors and hit the 128MB Worker memory limit.

```typescript
// workers/routes/chat-files.ts
// CRITICAL: Do NOT call c.req.formData() — it buffers the entire body in memory.
// Use c.req.raw.body (ReadableStream) for streaming pass-through.

import { Hono } from "hono";
import type { ParrotContext } from "../lib/mailbox";
import { requireEmployeeMailbox } from "../lib/mailbox";
import { getWorkspaceStub } from "../durableObject/workspace";

export const chatFilesRoute = new Hono<ParrotContext>();

// POST /api/chat/files?channel_id=:id — stream multipart upload to MM
chatFilesRoute.post("/", requireEmployeeMailbox, async (c) => {
  const channelId = c.req.query("channel_id");
  if (!channelId) return c.json({ error: "Missing channel_id" }, 400);

  const employee = c.var.employee;
  const stub = getWorkspaceStub(c.env);
  const tokenRow = await stub.getEmployeeToken(employee.employeeId);
  if (!tokenRow) return c.json({ error: "chat_not_provisioned" }, 503);

  const mmUrl = c.env.MATTERMOST_URL.replace(/\/$/, "");
  const upstreamResp = await fetch(
    `${mmUrl}/api/v4/files?channel_id=${channelId}`,
    {
      method: "POST",
      headers: {
        Authorization: `Bearer ${tokenRow.token}`,
        // Forward Content-Type exactly (includes multipart boundary)
        "Content-Type": c.req.header("content-type") ?? "multipart/form-data",
      },
      body: c.req.raw.body, // Stream directly — NO buffering
      // @ts-ignore: duplex required for streaming request body
      duplex: "half",
    },
  );

  const data = await upstreamResp.json().catch(() => null);
  return c.json(data, upstreamResp.status as 200);
});

// GET /api/chat/files/:fileId — proxy file download for inline preview
chatFilesRoute.get("/:fileId", requireEmployeeMailbox, async (c) => {
  const fileId = c.req.param("fileId");
  const employee = c.var.employee;
  const stub = getWorkspaceStub(c.env);
  const tokenRow = await stub.getEmployeeToken(employee.employeeId);
  if (!tokenRow) return c.json({ error: "chat_not_provisioned" }, 503);

  const mmUrl = c.env.MATTERMOST_URL.replace(/\/$/, "");
  const upstream = await fetch(`${mmUrl}/api/v4/files/${fileId}`, {
    headers: { Authorization: `Bearer ${tokenRow.token}` },
  });
  if (!upstream.ok) return c.json({ error: "file_not_found" }, upstream.status as 404);

  // Forward the file response with original Content-Type (so images render inline)
  return new Response(upstream.body, {
    status: 200,
    headers: {
      "Content-Type": upstream.headers.get("Content-Type") ?? "application/octet-stream",
      "Content-Disposition": upstream.headers.get("Content-Disposition") ?? "inline",
      "Cache-Control": "private, max-age=3600",
    },
  });
});
```

**1. File attach + drag-drop + inline preview:**

Upload flow:
- On file selection or drop, immediately POST to `/api/chat/files?channel_id=${activeId}` using `fetch()` directly (NOT via chatFetch — this is a multipart request, not JSON). Set the body to a FormData with the file appended as "files" field. The browser's native `fetch()` sets the correct multipart Content-Type + boundary automatically.
- On success, extract `file_infos[0].id` from the response and store it in a `pendingFileIds: string[]` state array.
- Show an upload progress indicator (pending state) and a preview thumbnail for images (use `URL.createObjectURL`).
- When the message is sent (form submit), include `file_ids: pendingFileIds` in the POST /api/chat/posts body alongside the message text.
- Clear `pendingFileIds` and any preview URLs after successful send.

In the message list, render inline image previews for posts that have `file_ids` or `metadata.files` in the MmPost structure. Add `metadata?: { files?: Array<{ id: string; name: string; mime_type: string; width?: number; height?: number }> }` to MmPost interface. For image mime types (image/*), render an `<img>` tag with `src="/api/chat/files/${file.id}"`. For other file types, render a download link.

**2. Global search bar:**

The WorkspaceShell header already has a disabled `<input type="search">` placeholder. Instead of enabling that one (it belongs to the shell), add a search input WITHIN the ChatPane main header area (the bar above the message list that currently shows the channel name and refresh button). Add a search icon button; clicking it toggles a search mode overlay.

Search mode:
- Shows a full-width search input inside the chat content area (replaces the message list view when active).
- On input change (debounced 400ms), POSTs to `/api/chat/search` with `{ terms: searchValue, team_id: bootstrap.data?.team.id }`.
- Results displayed as a scrollable list of matching posts: channel name + author + timestamp + message snippet (highlight the search term).
- Clicking a result jumps to that channel (sets activeChannelId to result.channel_id) and closes search mode.
- "Esc" or clicking X closes search mode.

**3. Emoji reaction picker:**

In the message hover action row (from Wave 1), add an emoji picker button (smiley face icon). Clicking it shows a small emoji quick-picker with ~20 common emoji (hardcoded list: 👍 ❤️ 😂 🎉 🔥 👀 🙏 💯 ✅ ❌ 😊 🤔 🚀 💡 ⚡ 😎 🤝 🙌 💪 ✨). Clicking an emoji calls POST /api/chat/reactions `{ post_id, emoji_name }` (use the emoji's text name, e.g. "thumbsup" for 👍).

To map emoji to names, use a small hardcoded object `EMOJI_NAMES = { "👍": "thumbsup", "❤️": "heart", ... }`.

Update MmPost interface: add `reactions?: Array<{ user_id: string; emoji_name: string }>` field. In the message list, show reaction counts below each message (grouped by emoji_name). Clicking an existing reaction adds/removes it (toggle: if employee already reacted with that emoji, call DELETE /api/chat/reactions; otherwise POST).

On mount for a channel, optionally pre-fetch reactions for the last 10 posts via getMmPostReactions — or more simply, include reactions in the posts query response by extending the backend to fetch reactions for each post. For simplicity in this wave, only show reactions if they are already included in post.reactions (MM may include them in the post object); do not make per-post reaction API calls on initial load.

**4. @mention autocomplete:**

In the message textarea's `onChange` handler, detect `@` followed by 1+ characters (use `/\B@(\w+)$/` regex on the current draft up to cursor position). When detected, filter the team member list (already available from Wave 2's /api/chat/team-members query — cache it in state) to show an autocomplete dropdown above the composer with matching usernames. Clicking a suggestion inserts `@username ` in the draft.

In the message list renderer, highlight @username patterns in message text using a simple regex replace: wrap matches of `/\B@(\w+)/g` in a `<span className="text-sky-600 font-medium">` span. If the mention matches the current employee's MM username (from `bootstrap.data?.me.username`), use a yellow highlight `bg-yellow-100` to indicate it's directed at them.
